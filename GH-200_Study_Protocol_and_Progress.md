## 3. Teaching protocol (agreed with the learner)
1. **Go domain by domain, in order 1 ‚Üí 5.** Cover **only gap/thin objectives**; skip what the courses already cover well.
2. Each domain is split into a few **major topics**. Each major topic is split into **small bits** (1‚Äì2 concepts each).
3. **Deliver one bit at a time. Then STOP and wait for the learner's explicit green light** before the next bit. The learner will question/challenge each bit ‚Äî engage, don't rush.
4. **At the end of each major topic** (not each domain ‚Äî that's too much), produce TWO things: (a) a **personalized recap MD** with the taught content *plus the outcomes of the learner's Q&A*, and (b) a **per-topic cheat-sheet MD** (`GH-200_Cheat_<code>_-_<name>.md`, mirroring the recap filename with `Topic`‚Üí`Cheat`) that collects ONLY the trap-grade "keepers" ‚Äî gotchas, easy-to-confuse contrasts, must-memorize specifics, and any item the learner explicitly flags "put this on the cheat sheet." **One cheat file per topic ‚Äî NOT a single running monolith** (switched to per-topic in Session 5: independent files remove the stale-base/clobbering risk and map 1:1 to recaps). *(Superseded 2026-06-24, post-teaching: the 14 recaps + 14 cheats were re-consolidated into two combined files ‚Äî see ¬ß8.)* Each cheat file starts with a one-line pointer to its recap. The cheat file is the terse last-week-review artifact; the recap is the fuller study note. The learner keeps both on their HD (see file-location convention in ¬ß8) ‚Äî they are NOT kept in the project; ask the learner to upload one when you need it. Tag keepers as you teach so they're easy to harvest at recap time.
5. **Quizzes are exam-style** ‚Äî multiple-choice / multiple-response, scenario-driven ("what does this config do?", "which option is correct?", "spot the mistake"), because GH-200 is **recognition-based, not free-recall** (the learner has IntelliSense/schema when authoring, and the exam is MCQ/multi-response). Use free-recall only sparingly, for the few things worth knowing cold. Offer a short quiz after a bit and/or after each major-topic recap. *(Exact GH-200 item-type mix not yet verified against the official exam page ‚Äî confirm if precision needed.)*
6. **Practice exams come last**, after the gaps are studied ‚Äî they are the real diagnostic and will also expose retention holes in already-covered material. Anchor on the **free official Microsoft Learn practice assessment**; supplement with others from the provider list.
7. **Timestamp every edit** (convention set Session 8). The header carries a `**Last updated: YYYY-MM-DD HH:MM UTC**` line; bump it to the current time whenever this file is edited, so a stale project copy is instantly distinguishable from a fresh one. (Same-day sessions happen, so use date+time, not date alone.)
8. **Formatting (convention refined Session 8).** In **chat**: fenced blocks **small only**. Anything **big/wide** (esp. multi-column tables) ‚Üí put it in a downloadable **`.md` file** as real (unfenced) markdown tables. **Avoid** raw markdown tables in chat (render as a mess) and **avoid** big fenced blocks in chat (wrap badly). In **files**: real markdown tables + normal fenced code blocks are fine (they render in a reader). Supersedes the older Session-2 "fence all tricky markdown" rule.
